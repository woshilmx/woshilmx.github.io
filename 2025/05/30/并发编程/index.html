如何正确停止一个线程

线程启动

  • 使用 Thread 类的 start() 方法启动线程。
  • 在 run() 方法中定义要执行的任务。

正确停止线程

  • Java 不强制停止线程的原因：

    • Java 不提供强制停止线程的方法，以避免在强制停止时出现数据不完整或其他安全问题。
    • 停止线程的方式应该是通知线程，让它决定如何结束任务。

  • 正确的停止线程方法：

    • 使用 interrupt() 方法请求线程中断。
    • 线程在 run() 方法中定期检查 Thread.currentThread().isInterrupted()，以判断是否需要停止。

3. 处理中断异常

  • 睡眠期间的中断：

    • 线程在 Thread.sleep()、wait() 等阻塞方法中被中断时，线程会感知中断信号并抛出 InterruptedException 异常。
    • 需要在 catch 语句中处理异常，或者将异常抛给调用方。

  • 最佳实践：

    • 在方法签名中声明 throws InterruptedException。
    • 在 catch 语句中重新设置中断标记 Thread.currentThread().interrupt()，以便后续代码能感知到中断。

4. 错误的停止方法

  • 已废弃的方法：

    • stop(): 强制停止线程，可能导致数据不一致。
    • suspend() 和 resume(): 可能导致死锁，因为 suspend() 不释放持有的锁。

  • 使用 volatile 标记位：

    • volatile 标记位用于表示线程停止的请求，但在某些场景下（如线程被阻塞时），可能无法及时感知中断信号。

  • 生产者/消费者模式问题：

    • 在使用 volatile 标记位的情况下，如果生产者线程在阻塞（如 BlockingQueue.put()）时无法及时响应中断。

总结

  • 正确停止线程的方法是使用 interrupt() 来请求中断，而不是强制停止线程。
  • 在处理 InterruptedException 异常时，应正确处理或重新中断，以确保线程能够感知中断信号。
  • volatile 标记位虽然可以用于线程停止，但在某些情况下（如线程阻塞）可能不够可靠。

状态

  1. New（新创建）
  2. Runnable（可运行）
    1. 对应的是操作系统的准备状态与运行状态
  3. Blocked（被阻塞）

wait、notify、notifyAll 方法的使用注意事项

为什么 wait 方法必须在 synchronized 保护的同步代码中使用？

在多线程编程中，wait 方法的正确使用必须在 synchronized 保护的同步代码块中。源码注释强调了 wait 方法应该放在 while 循环中，并在调用 wait 方法之前，必须先持有对象的监视器锁（即 synchronized 锁）。这是为了确保线程的操作是原子的，不会被其他线程打断，从而避免线程不安全问题。

为什么 wait/notify/notifyAll 被定义在 Object 类中，而 sleep 定义在 Thread 类中？

  1. 监视器锁的对象级别：每个对象都有一个监视器锁（monitor），锁是对象级别的，而不是线程级别的。wait/notify/notifyAll 是对象级别的操作，因此定义在 Object 类中最合适。
  2. 多锁管理的需要：一个线程可以持有多把锁，如果 wait 方法定义在 Thread 类中，管理和使用多个锁将变得复杂和不明确。通过对象来操作锁，能更清晰地表达线程在等待哪个对象的锁。

wait/notify 和 sleep 方法的异同

相同点：

  1. 都可以让线程阻塞。
  2. 都可以响应 interrupt 中断，在等待过程中抛出 InterruptedException 异常。

不同点：

  1. 使用环境：wait 必须在 synchronized 保护的代码中使用，而 sleep 没有这个要求。
  2. 锁的释放：在同步代码中，sleep 方法不会释放监视器锁，而 wait 方法会主动释放锁。
  3. 恢复方式：sleep 方法需要指定时间，到期后自动恢复；而无参的 wait 方法会永久等待，直到被中断或被唤醒。
  4. 定义位置：wait/notify 是 Object 类的方法，而 sleep 是 Thread 类的方法。
